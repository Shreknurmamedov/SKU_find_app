# Видео-аудит SKU: пайплайн

Двухэтапная схема определения присутствующих SKU по видео полки. Главная
бизнес-цель — не количество физических коробок/инструментов, а список
уникальных моделей/артикулов, которые представлены в торговой точке.

Детектор не знает брендов — он только находит товар и собирает кропы; бренд,
модель и артикул определяются по кропу через OCR, каталог и эталонные фото.

```
видео ─► [1] детекция+трекинг ─► [2] re-ID ─► [3] распознавание ─► отчет
         (ByteTrack: 1 трек        (склейка       (OCR + visual)      brand/
          = 1 кандидат)             повторных                        category/
                                    проходов)                         model/article
```

## Этап 1 — детектор товара (`ml/train_detector.py`, `ml/track_video.py`)

- Один класс `product`. Датасет собирается из старой 2-классовой разметки:
  `python3 ml/build_product_dataset.py` → `ml/datasets/sku_products`.
- Обучение: `python3 ml/train_detector.py` (yolo11n, MPS). Веса →
  `weights/product_det.pt`. Текущий baseline: mAP50 ≈ 0.44.
- Трекинг нужен не ради финального количества, а чтобы собрать хорошие кропы без
  повторов одного и того же объекта между кадрами. Для каждого трека сохраняется
  лучший кроп и несколько запасных кандидатов для OCR/визуального поиска.

## Этап 2 — re-ID (`ml/reid_merge.py`)

Склеивает треки, возникшие при **повторном проходе** камеры. Сливает только если:
треки не пересекаются во времени, между ними реальный временной разрыв
(`min_gap_frames`) и кропы визуально похожи (`sim_threshold`, по умолчанию 0.96).
Консервативно намеренно: на одиночном проходе рядом стоящие одинаковые товары
могут остаться разными объектами, но после SKU-recognition они схлопываются в
один уникальный `sku_id`. Эмбеддинги — из backbone YOLO.

## Этап 3 — распознавание (`ml/sku_recognize.py`)

OCR кропа (easyocr ru+en) с перебором поворотов → матчинг с
`data/catalog/own_products.csv`: сначала по артикулу/коду модели, затем по бренду.
Бренд/модель/группа берутся из строки каталога. Если OCR не дал точный SKU, для
лучших объектов включается fallback по эталонным фото через
`data/catalog/reference_index_yolo11n.npz` + `reference_index_yolo11n.jsonl`.

Артикулы матчятся строго: `73/7/2/6` не должен совпадать как подстрока внутри
`73/7/2/26`. Модельные коды допускают узкую OCR-нормализацию (`5P`/`CP` → `SP`,
`ЭТ/ЛИ` ↔ `ET/LI`, пропущенная начальная `G/E` в `GET/ET`), чтобы не терять
случаи вроде `SP-3,7 Lite` и `GET-20M-2Li`.

Эталонный индекс собран из `data/catalog/reference_dataset_all/training_images.csv`
(14 441 фото: Вихрь, Ресанта, Huter, Зубр, Fubag, Интерскол, Patriot). Визуальный
поиск используется только при высоком сходстве и достаточном отрыве от второго
кандидата, потому что похожие коробки разных брендов иногда дают близкие
эмбеддинги. OCR остается главным сигналом, visual — осторожный второй сигнал.

Для скорости `ml/audit_video.py` делает быстрый OCR обычных объектов в одном
повороте, а enhanced OCR + полный набор поворотов + visual fallback запускает
только для `--ocr-retry-objects` лучших детекций. Среди `crop_candidates` аудит
сначала выбирает более OCR-friendly crop по резкости, edge-density и числу
текстоподобных компонент, а не просто самый уверенный detector-box.

## Запуск аудита (`ml/audit_video.py`)

```bash
python3 -m ml.audit_video --video "ТТ Пэкстрой/IMG_8886.MOV" \
    --weights weights/product_det.pt --device mps
# --reuse-tracks  пропустить детекцию, если tracks.json уже есть
# --ocr-retry-objects 10  сколько лучших объектов читать с enhanced OCR + visual fallback
```

Отчет → `reports/audit_<видео>.{json,md}`: найденные уникальные SKU/артикулы,
разбивка по брендам/группам/моделям и раздел «спорные зоны» с кропами, где
товар виден, но SKU не прочитан.

Object-level поля (`candidate_objects`, `sku_evidence_objects`,
`needs_review_objects`) остаются диагностикой качества съемки и OCR. Финальный
ответ для бизнеса — `sku_presence` и `totals.unique_skus`.

Прогнать на нескольких видео сразу: `scripts/run_final_audit.sh`.

## Как мерить качество

- Главная метрика: SKU recall — сколько ожидаемых моделей/артикулов найдено.
- Вторая метрика: SKU precision — сколько найденных SKU действительно есть на
  полке.
- Object count error больше не является бизнес-метрикой: пять одинаковых
  инструментов одной модели должны дать один найденный SKU.

```bash
cp data/eval/sku_presence.example.csv data/eval/sku_presence.csv
# заполнить expected_sku_ids через |, используя sku_id из каталога
python3 -m ml.evaluate_sku_coverage \
  --truth data/eval/sku_presence.csv \
  --reports-dir reports
```

## Ограничения и путь к высокой точности

- Детектор всё ещё должен видеть товары достаточно полно, чтобы собрать хотя бы
  один хороший кроп на каждый SKU. Но ему не нужно идеально считать каждую
  физическую штуку.
- Визуальный индекс уже подключен, но не является абсолютной истиной: для сложных
  витрин все равно нужен ручной `sku_presence.csv` на holdout-видео и проверка
  спорных зон. Чем больше эталонных фото одной модели с разных ракурсов, тем
  полезнее fallback там, где OCR не читает текст.
